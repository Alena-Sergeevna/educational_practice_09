# Задание: техническое задание на разработку клиентской части веб-приложения

**Результат:** один документ **ТЗ** (оформление — Word). **Код клиента не пишется** — только проектная документация.

---

## 1. Цель работы

Научиться по **контракту REST API** и предметной области формулировать **техническое задание** на **клиентскую часть** веб-приложения (браузер): экраны, сценарии, роли, интеграция с сервером, требования к ошибкам и безопасности на стороне клиента.

---

## 2. Контекст (роли в учебном процессе)

В репозитории **IT Company Practice API** реализован **сервер (бэкенд)** на Laravel. Ваш документ ТЗ описывает **будущий фронтенд** (SPA или классическое MPA — вы фиксируете это в ТЗ как допущение), который обращается к API по `HTTP(S)` с JSON и заголовком авторизации.

---

## 3. Варианты задания

Каждый обучающийся выполняет ТЗ **только для своего варианта** (домена API). Номер варианта выдаёт преподаватель.

| Вариант | Ключ роли в API | Содержание клиентского приложения (предметно) |
|--------|-----------------|-----------------------------------------------|
| **1. Projects** | `projects` (и `admin`) | Проекты, этапы (milestones), задачи (tasks); списки, карточки, связи; аналитика раздела Projects. |
| **2. Org** | `org` (и `admin`) | Отделы, сотрудники, команды; привязка сотрудника к команде; аналитика раздела Org. |
| **3. Assets** | `assets` (и `admin`) | Оборудование (hardware), лицензии ПО, выдача лицензий сотрудникам; аналитика раздела Assets. |
| **4. Tickets** | `tickets` (и `admin`) | Категории заявок, заявки (records), комментарии; аналитика раздела Tickets. |
| **5. Hiring** | `hiring` (и `admin`) | Вакансии, кандидаты, отклики, интервью; аналитика раздела Hiring. |

Роль **`admin`** в ТЗ нужно описать: полный доступ ко **всем** разделам API (в рамках учебного сценария). Роль варианта — доступ **только** к своему префиксу (как на сервере).

## 4. Обязательная структура документа ТЗ

1. Титульный лист: условное название системы, вариант, ФИО, группа, дата.  
2. Введение: назначение документа; что разрабатывается (клиент к существующему API).  
3. Основания для разработки: ссылка на репозиторий API, на файл OpenAPI (см. раздел 6); указание, что сервер уже задан, клиент — новый продукт.  
4. Цели и задачи клиента в рамках варианта (2–5 **измеримых** целей).  
5. Требования к среде: целевые браузеры, минимальная ширина экрана; при необходимости — HTTPS для продуктива (без детализации настройки сервера).  
6. Роли пользователей **на клиенте** (что видит и что может делать каждый тип), согласованные с ролями API.  
7. **Функциональные требования:**  
   - дерево **экранов** и навигация;  
   - для каждого экрана: назначение, состав элементов UI, **какие HTTP-запросы** к API (метод, путь, основные query/body), валидации на клиенте;  
   - сценарии CRUD там, где это предусмотрено API;  
   - для **списков**: явно указать использование **поиска** (`q`), **сортировки** (`sort`, `direction`), **пагинации** (`page`, `per_page`) — там, где это описано в OpenAPI для соответствующих ресурсов;  
   - подраздел **«Аналитика»**: какие показатели, с каких URL аналитики (см. таблицы в разделе 7), в каком виде представляются пользователю.  
8. **Нефункциональные требования:** индикация загрузки, обработка ошибок ответов API (например 401, 403, 422, 5xx) — что показывать пользователю.  
9. **Безопасность на клиенте:** где хранить токен (обосновать выбор), выход из системы, поведение при 401 / невалидном токене.  
10. **Модель данных на клиенте:** сущности и связи (родитель–потомок по варианту); таблица или диаграмма (допускается Mermaid).  
11. **Ограничения и допущения:** что не входит в объём (мобильное приложение, офлайн, функции без поддержки в API и т.д.).  
12. **Критерии приёмки:** проверяемые утверждения («пользователь с ролью … может … через экран …»).

Ориентир по объёму: **8–15 страниц** A4 при обычном шрифте — за счёт полноты, а не «воды».

## 6. Материалы

### 6.1. Описание API

- Учебный **REST API** на **Laravel**, тематика **IT-компания**.  
- Базовый префикс маршрутов API: **`/api/v1`**.  
- **Аутентификация:** Laravel Sanctum.  
  - Вход: **`POST /api/v1/auth/login`**, тело JSON: поля **`email`**, **`password`**.  
  - Ответ содержит токен (конкретное имя поля и форма ответа — по **OpenAPI** или по примеру в Swagger при работающем сервере).  
  - Защищённые методы: заголовок **`Authorization: Bearer <token>`**.  
  - После входа: **`POST /api/v1/auth/logout`**, **`GET /api/v1/auth/me`** (оба под защитой `auth:sanctum`).  
- **Роли** пользователя в базе: `admin`, `projects`, `org`, `assets`, `tickets`, `hiring`.  
  - **`admin`** — все разделы ниже.  
  - Остальные роли — только **свой** раздел (см. таблицу вариантов в разделе 3).  
- **Демо-аккаунты** (после выполнения миграций и сидов на сервере; пароль у всех одинаковый):

| Email | Роль / комментарий |
|-------|---------------------|
| `admin@company.test` | `admin` — все разделы |
| `projects@company.test` | только Projects |
| `org@company.test` | только Org |
| `assets@company.test` | только Assets |
| `tickets@company.test` | только Tickets |
| `hiring@company.test` | только Hiring |

**Пароль для всех демо-аккаунтов:** `password` 

### 6.2. Контракт API (OpenAPI 3)

В репозитории хранится сгенерированная спецификация:

- **`/docs/api-docs.json`**

Используйте её как **единственный обязательный источник** по полям запросов/ответов, кодам ошибок, query-параметрам для каждого метода.

### 6.3. Общие параметры списков

Для **индексных** (списочных) эндпоинтов в типичном случае доступны (если явно указано в OpenAPI для конкретного маршрута):

- **`q`** — текстовый поиск по полям ресурса;  
- **`sort`**, **`direction`** (`asc` / `desc`) — сортировка;  
- **`page`**, **`per_page`** — пагинация;  
- дополнительные фильтры (например `status`, `department_id`) — **только если** они описаны для данного маршрута в `api-docs.json`.

В ТЗ нужно **явно** указать, для каких экранов/списков вы требуете поддержку этих параметров на клиенте.

## 7. Сводка маршрутов по вариантам (пути относительно `/api/v1`)

Все перечисленные ниже маршруты (кроме входа и корня) требуют **`Authorization: Bearer <token>`** и допускаются только для ролей, указанных в разделе 3.

### 7.1. Публично (без токена)

| Метод | Путь | Назначение |
|--------|------|--------------|
| GET | `/` (в коде это `GET /api/v1`) | Подсказки: вход, демо-аккаунты, ссылки на разделы (на работающем сервере) |
| POST | `/auth/login` | Получение токена |

### 7.2. Общие (с токеном)

| Метод | Путь | Назначение |
|--------|------|-------------|
| POST | `/auth/logout` | Выход |
| GET | `/auth/me` | Текущий пользователь |

### 7.3. Вариант 1 — Projects (роли: `admin`, `projects`)

Префикс ресурсов: **`/projects`** (вложенные — под конкретным проектом).

**Аналитика (GET):**

| Путь |
|------|
| `/projects/analytics/overview` |
| `/projects/analytics/tasks-by-priority` |
| `/projects/analytics/task-workload` |

**REST (стандартный набор `apiResource` Laravel):**

| Ресурс | Базовый путь | Примечание |
|--------|----------------|------------|
| Проекты | `/projects` | CRUD + список |
| Этапы | `/projects/{project}/milestones` | CRUD + список |
| Задачи | `/projects/{project}/tasks` | CRUD + список |

Методы для каждого `apiResource`: `GET` список и `GET` один элемент, `POST`, `PUT/PATCH`, `DELETE` — уточнить по OpenAPI (идентификаторы в пути, тела запросов).
